# API Gateway — Developer Portal & Documentation

> Research document for developer experience, API documentation, and self-service integration.

---

## Key Questions

1. **What documentation do API consumers (partners, internal teams) need?**
2. **How do we keep documentation in sync with the actual API?**
3. **What self-service capabilities should the developer portal provide?**
4. **How do we manage API keys and access credentials?**
5. **What's the interactive testing experience (Postman, Swagger UI, custom)?**

### Developer Portal Features

```typescript
interface DeveloperPortal {
  documentation: DocumentationSystem;
  interactiveExplorer: APIExplorer;
  sdkManagement: SDKConfig;

interface DocumentationSystem {

  autoGenerated: boolean;

interface Guide {
```

## Open Problems

1. **Documentation freshness** — Auto-generated docs from OpenAPI are always current but lack narrative. Hand-written guides are better but drift from the API.

2. **Partner onboarding friction** — Getting API access should take minutes, not days. Self-service key creation with instant access is ideal but has security implications.

3. **Webhook reliability** — Consumer endpoints may be down. Retry logic helps but eventually, undelivered events need a dead-letter queue and reconciliation.

4. **SDK generation** — Generating client SDKs from OpenAPI spec saves partner time but maintaining generated SDKs across versions is costly.

5. **Rate limit transparency** — Consumers need to understand their usage relative to limits without manually tracking response headers.

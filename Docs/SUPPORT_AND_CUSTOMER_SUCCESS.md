# Support and Customer Success

**Date**: 2026-04-14
**Purpose**: How to handle support as a solo builder without burning out

---

## The Solo Builder Support Dilemma

> "Every support request is context switch. Every context switch is progress lost."

But: Early customers = friends. Treat them well or they leave.

---

## Support Philosophy

**Principle**: Fast, personal response + boundaries

**Not**: 24/7 availability (you're not a big company)

---

## Support Channel Strategy

### Primary: Email

| Pros | Cons |
|------|------|
| Async, works on your schedule | Can feel slow |
| Searchable history | Less personal |
| Easy to template | Spam folder risk |

**Use for**: Non-urgent questions, bugs, feature requests

---

### Secondary: WhatsApp (for VIPs)

| Pros | Cons |
|------|------|
| Fast, personal | Interruptions |
| Builds relationship | Expectations of 24/7 |
| Screenshots easy | Hard to track |

**Use for**: Design partners, paid pilots, urgent issues only

**Give them your number, but set expectations**:
> "You can WhatsApp me for urgent stuff 9am-7pm IST. For everything else, email is better."

---

### Tertiary: In-app feedback widget

| Pros | Cons |
|------|------|
| Contextual | Another thing to build |
| Low friction | Noise |
| Always available | Can be ignored |

**Use for**: Simple "was this helpful?" buttons, bug reports

**Don't build yet** — use email first.

---

## Response Time SLA (Solo-Friendly)

| Tier |承诺 | Reality |
|------|------|---------|
| **Critical** (system down) | 2 hours | Fix immediately |
| **Urgent** (can't work) | 4 hours | Same day |
| **Normal** (questions) | 24 hours | Usually faster, but 24h is okay |
| **Low** (features) | 3 days | When you can |

**Communicate this clearly**: "I respond within 24 hours, usually faster."

## Onboarding: Make It Self-Service

### First-Time Experience

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  NEW AGENCY ONBOARDING FLOW                                                     │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  1. Sign up (Clerk magic link)                                                  │
│       ↓                                                                         │
│  2. Welcome page: "Let's process your first trip"                              │
│       ↓                                                                         │
│  3. Choose:                                                                    │
│      - "I have a trip I'm working on now" → Go to intake                       │
│      - "Show me how it works" → Example trip                                   │
│      - "Upload past conversation" → See what it extracts                       │
│       ↓                                                                         │
│  4. First success: "Great! Here are your options."                             │
│       ↓                                                                         │
│  5. Celebrate: "You just saved 30 minutes."                                    │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### Documentation (Keep It Minimal)

**You need**:
1. **Getting Started** (1 page): Sign up → first trip
2. **FAQ** (10-20 questions): Common issues
3. **Video walkthrough** (5 min): Show, don't tell

**Nice to have** (later):
- Detailed feature guides
- API documentation
- Advanced tips

### Canned Responses (Save Time)

**Create templates for**:
- Welcome email
- "How do I get started?"
- Common error messages
- Feature request acknowledgement
- Churn feedback request

## Customer Success: Prevent Problems

### Proactive Check-ins

| Timing | Action |
|--------|--------|
| **Day 1** | Welcome email + "book a 15-min onboarding call" |
| **Day 3** | "How was your first trip?" |
| **Week 2** | "Any questions? Free 15-min call if needed." |
| **Month 1** | "How's it going? What would make this better?" |
| **Quarterly** | For active users: "You've processed X trips. Here's what's coming." |

### Power User Recognition

When someone uses it a lot:
> "Hey, noticed you've processed 20 trips this month! You're a power user. Want a free call to discuss advanced features?"

They become:
- Beta testers for new features
- References for case studies
- Word-of-mouth champions

## When Customers Leave (Churn Analysis)

### The Exit Interview

When someone cancels or goes dormant:

> "I noticed you haven't used the tool in a while. No worries if it's not a fit, but I'd love to learn: What was the main reason?"

**Categories to track**:
- Not useful enough
- Too expensive
- Too complex to learn
- Missing a specific feature
- Just busy (might return)

**Every churn is learning**. Document reasons. Fix common ones.

## Setting Boundaries (Critical for Solo)

### Your Hours

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  MY SUPPORT BOUNDARIES                                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Response time: Within 24 hours (usually faster)                                │
│  Working hours: 10am-7pm IST (Mon-Sat)                                          │
│  Sunday: Offline (emergencies only)                                             │
│  WhatsApp: Design partners only                                                │
│  Phone: By appointment only                                                     │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### Communicate This

In your email footer, FAQ, and onboarding:
> **Support**: email@yourdomain.com. Responses within 24 hours (Mon-Sat).

### The "Emergency" Definition

What counts as emergency:
- System is down for everyone
- Data loss
- Security breach

What doesn't count:
- "I need this now because I procrastinated"
- "I'm confused about how to use this"
- "Can we hop on a call?"

## Measuring Support Health

| Metric | How to track | Target |
|--------|--------------|--------|
| **Response time** | Email timestamps | < 24 hours |
| **Resolution time** | Time to close ticket | < 48 hours |
| **FAQ usage** | Page views | Growing |
| **Repeat questions** | Identify patterns | Add to FAQ |
| **Churn reasons** | Exit interviews | Track top 3 |

---

## When to Hire Support Help

| Trigger | Action |
|---------|--------|
| > 20 tickets/week | Hire part-time contractor |
| > 50 tickets/week | Full-time support hire |
| You're spending > 20% time on support | Scale or hire |

**Before hiring**: Improve product + documentation to reduce tickets.

---

## Summary

**Channels**: Email (primary), WhatsApp (VIPs only)

**SLA**: 24-hour response, critical < 4 hours

**Onboarding**: Self-service with optional 15-min call

**Boundaries**: Clear hours, communicate them

**Philosophy**: Fast, personal — but not 24/7

**Every support request is**: Learning opportunity + relationship building

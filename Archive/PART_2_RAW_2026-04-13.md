## Core response themes captured

1. Separate live path from optimization path
- Live path should be deterministic/controlled:
  - context assembly
  - pre-router analysis
  - prompt composition from reusable blocks
  - specialist execution
  - verifier pass
  - memory update
- Offline path should use autoresearch-style loops with measurable evals.

2. “Compiler pipeline” model (not one monolithic prompt)
- Normalize context into structured packet
- Route/classify using cheap model first
- Compose prompt from prompt registry (not fresh prompt generation each turn)
- Run selected agent
- Verify policy/quality/tool correctness

3. Karpathy autoresearch usage guidance
- Valid fit: offline optimization of router prompt, prompt packs, context packing, thresholds
- Invalid fit: per-turn live prompt rewriting / taxonomy drift / agent explosion
- Principle transferred from autoresearch: constrained search space + fixed eval harness + measurable metric.

4. Two-loop architecture
- Online loop for production replies
- Offline loop for continuous improvement using logged runs and eval datasets

5. Required implementation contracts
- Context packet schema
- Router output schema (JSON control output)
- Prompt registry profiles
- Specialist response schema
- Verifier output schema
- Logging schema (versioned run metadata)
- Event model and state transitions

6. Recommended taxonomy discipline
- Start coarse-grained in v1 (e.g., support/sales/billing/etc.)
- Avoid early micro-agent fragmentation
- Split only when error clusters justify it

7. Suggested rollout order
- deterministic skeleton
- eval harness + labeled cases
- offline optimization loop
- selective specialization
